# Synthesis & Recommendations: Agent Teams Integration with MPM
## Issue #290 — Initial Investigation Results

**Date:** 2026-03-20
**Author:** PM (Team Lead), synthesizing findings from 3 parallel investigators
**Status:** COMPLETE — Ready for decision

---

## Executive Summary

Three investigators examined GitHub issue #290 (integrating Anthropic's Agent Teams with MPM orchestration) from complementary perspectives: capabilities research, architecture analysis, and devil's advocacy. **All three converge on the same core finding:**

> The integration is architecturally sound in concept but has **two critical blockers** that must be resolved via proof-of-concept before any production work begins. The existing MPM parallel execution model (`run_in_background` + `isolation: "worktree"`) already covers ~80% of the proposed use cases. Agent Teams' unique value — real-time peer coordination — addresses the remaining ~20% but introduces significant risk to MPM's verification integrity.

**Recommendation: Proceed to Phase 0 (proof-of-concept) only. Do not begin integration coding.**

## Architectural Clarification: How Context Flows

**Important:** The context injection mechanism must be precisely understood to assess fix paths.

```
MPM (Python layer)
  |
  |-- Launches Claude Code with CLI argument including PM_INSTRUCTIONS.md
  |     (PM_INSTRUCTIONS.md is generated by MPM based on available agents & skills)
  |
  v
Claude Code instance (Team Lead / PM)
  |-- Has FULL MPM context: PM_INSTRUCTIONS.md, agent catalog, skills, circuit breakers
  |
  |-- Team Lead spawns teammates via Agent tool (team_name parameter)
  |     Claude Code itself launches these teammate processes
  |     MPM does NOT control this launch -- Claude Code does
  |
  v
Teammate instances (launched by Claude Code, NOT by MPM)
  |-- Receive: CLAUDE.md (project instructions), team tools (SendMessage, TaskList, etc.)
  |-- DO NOT receive: PM_INSTRUCTIONS.md, BASE_AGENT.md, agent templates, skills, circuit breakers
```

**Key insight:** The team lead has everything. The gap is exclusively at the **Claude Code teammate spawning boundary** — MPM controls the initial Claude Code launch but has no control over how Claude Code subsequently spawns teammates.

## Key Findings (Cross-Perspective Consensus)

### Finding 1: The Teammate Context Gap Is Real and Critical
**Consensus: ALL THREE investigators confirm this as the #1 blocker.**

When Claude Code spawns teammates (on behalf of the team lead), they receive the project's CLAUDE.md but **not** MPM's PM_INSTRUCTIONS.md, agent templates, BASE_AGENT.md, circuit breakers, QA verification gates, or skill injections. MPM generates PM_INSTRUCTIONS.md dynamically based on available agents and skills, and passes it to Claude Code at launch via CLI argument — but this injection only happens at the **initial** launch. Subsequent teammate launches are performed by Claude Code itself, bypassing MPM entirely.

Teammates are therefore generic Claude instances that violate every MPM protocol by default.

- **Capabilities Researcher**: Verified by examining live Agent Teams session — teammate tools confirmed, MPM context confirmed absent.
- **Architecture Analyst**: Traced the root cause — Claude Code's native spawning (Path B) bypasses MPM's Python `AgentLoader` entirely. The three-tier agent resolution system only activates for Task tool delegations (Path A).
- **Devil's Advocate**: Rated CRITICAL. Noted that BASE_AGENT.md line 185 explicitly states "do not use Agent Teams inside mpm PM delegation" — someone already considered and rejected this.

**The two spawning paths:**
```
Path A (MPM-controlled launch):
  MPM Python -> CLI arg with PM_INSTRUCTIONS.md -> Claude Code -> FULL MPM context

Path B (Claude Code teammate spawn — THE GAP):
  Team Lead -> Agent(team_name) -> Claude Code spawns process -> CLAUDE.md only -> NO MPM context
```

**Fix paths this clarification reveals:**
1. **Prompt injection by team lead**: Team lead (which HAS full context) includes BASE_AGENT/agent-specific instructions in the Agent tool's `prompt` parameter when spawning teammates. No MPM code changes needed — purely a PM_INSTRUCTIONS.md behavioral change.
2. **Hook-based injection**: MPM intercepts teammate spawn events via hooks and injects context. Requires a hook that fires BEFORE teammate execution (currently only TeammateIdle fires AFTER).
3. **Deploy agent definitions to .claude/agents/**: If MPM pre-deploys agent templates (with BASE_AGENT prepended) to `.claude/agents/`, Claude Code may pick them up when spawning teammates with matching `subagent_type`. Requires testing.
4. **Claude Code API change**: Request Anthropic add a mechanism for parent context inheritance or custom system prompts for teammates. Longest path but cleanest solution.

### Finding 2: Agent Teams Value Proposition — Broader Than Initially Assessed

**Post-investigation clarification from project owner:** The primary motivations for Agent Teams integration are:

1. **Independent context windows** — Each teammate operates in its own context window, preventing the team lead's (PM's) context from being consumed by delegated work. In the current Task tool model, agent results return into PM's context, progressively saturating it during complex multi-phase workflows.

2. **Per-teammate model selection** — Teammates can run on different models than the team lead (e.g., team lead on Opus for orchestration, research teammates on Sonnet, ops on Haiku). This enables fine-grained cost optimization and capability matching at the individual agent level.

3. **Real-time peer coordination** — Teammates can communicate directly, share discoveries mid-task, and redirect effort without round-tripping through the team lead.

**Revised capability comparison:**

| Capability | Current MPM (Task tool) | Agent Teams |
|------------|------------------------|-------------|
| Parallel agent execution | `run_in_background: true` | Also supported |
| File isolation | `isolation: "worktree"` | Also supported |
| Independent context windows | Partial — agent runs in own context, but **results return into PM context** | **Full — teammates have independent context; only targeted messages return to team lead** |
| Per-agent model selection | `model` param on Agent tool (available) | `model` param on Agent tool (same mechanism) |
| PM context preservation | **No** — each completed task adds to PM's context load | **Yes — team lead receives only summaries/messages, not full agent output** |
| Real-time peer coordination | **No** | **Yes** |
| Shared task list | **No** | **Yes** |
| Mid-task redirection | **No** | **Yes** |

**Revised assessment:** The investigation initially concluded Agent Teams' unique value was "real-time peer coordination only" (~20% gap). With the context window and model selection motivations clarified, the value proposition is substantially stronger:

- **Context window relief** is an architectural necessity for complex multi-phase workflows, not a nice-to-have. PM context saturation is a real operational problem in long sessions.
- **Model flexibility** aligns with MPM's existing cost-conscious model routing (PM_INSTRUCTIONS.md Model Selection Protocol) and extends it to team-level granularity.
- **Peer coordination** remains valuable but is now the third motivation, not the first.

This reframing shifts the question from "is the 20% gap worth the risk?" to "is context window management and cost optimization worth the risk?" — which is a stronger case for proceeding.

### Finding 3: Circuit Breaker Integrity Is at Risk
**Consensus: Architecture Analyst and Devil's Advocate both rate this CRITICAL.**

MPM's 13 circuit breakers are PM-centric — they work because all work flows through the PM. Peer-to-peer messaging creates paths that bypass PM oversight:

| Circuit Breaker | Bypass Risk | Mechanism |
|-----------------|-------------|-----------|
| CB#3: Unverified Assertions | HIGH | Teammates assert to each other without PM verification |
| CB#5: Delegation Chain | HIGH | P2P coordination skips PM's delegation tracking |
| CB#8: QA Verification Gate | HIGH | Team-internal "completion" without PM QA gate |
| CB#4: File Tracking | MEDIUM | Multiple agents committing without PM tracking |
| CB#10: Delegation Failure Limit | MEDIUM | PM can't detect failure loops in teammate P2P |

### Finding 4: Existing Hook Infrastructure Is a Foundation
**Consensus: Capabilities Researcher and Architecture Analyst both confirm.**

MPM already handles Agent Teams events passively:
- `TeammateIdle` -> `handle_teammate_idle_fast()` -> Dashboard emission
- `TaskCompleted` -> `handle_task_completed_fast()` -> Dashboard emission
- Version-gated to Claude Code >= 2.1.47 with migration for older versions

This observability layer is a foundation but does **not** include instruction injection, workflow triggering, or verification enforcement.

### Finding 5: Integration Difficulty Is Asymmetric
**Source: Architecture Analyst's difficulty ratings.**

| Integration Point | Difficulty | Prerequisite? |
|-------------------|-----------|---------------|
| Hook handler extension (dashboard) | EASY | No |
| Context gap resolution (agent_processing_step.py) | MEDIUM | **P0 — blocks everything** |
| Session memory sharing (Option C: team-lead broker) | MEDIUM | No |
| File tracking for parallel commits | MEDIUM | No |
| Team lead role definition (new instruction set) | HARD | **P0 — blocks safe usage** |
| QA aggregation protocol | HARD | P1 — blocks parallel QA |
| Circuit breaker adaptation for teams | HARD | **P0 — blocks safe usage** |

Three of the seven integration points are P0 prerequisites, and two of those three are rated HARD.

## Risk Assessment

### Critical Risks (Must resolve before proceeding)

| Risk | Severity | Mitigation | Owner |
|------|----------|------------|-------|
| Teammates lack MPM context | CRITICAL | Prototype prompt injection (full BASE_AGENT in task prompt) | Phase 0 |
| Circuit breakers bypassed via P2P | CRITICAL | Write formal Team Circuit Breaker Protocol | Phase 0 |

### High Risks (Must address during implementation)

| Risk | Severity | Mitigation |
|------|----------|------------|
| Experimental API instability | HIGH | Strict fallback to `run_in_background` when flag absent |
| Cost multiplication (3x for parallel teams) | MEDIUM (revised) | Mitigated by per-teammate model selection (Haiku teammates for deterministic tasks) and by context window savings (PM doesn't need to re-read returned results). Net cost may be neutral or lower for complex workflows. |
| Complexity budget exceeded | HIGH | Agent Teams must be invisible abstraction — PM sees "team result" as atomic |

### Medium/Low Risks (Manageable)

| Risk | Severity | Mitigation |
|------|----------|------------|
| Alternative approaches cover 80% of use cases | MEDIUM | Only use Agent Teams for the 20% that needs real-time coordination |
| UX fragmentation (two coordination models) | LOW | Make Agent Teams invisible to users — PM decides internally |

## Recommended Approach

### Phase 0: Proof-of-Concept (RECOMMENDED NEXT STEP)
**Goal:** Validate the two critical blockers can be solved. No production code.

**Deliverable 1: Teammate Context Injection Test**

Three fix paths to test (in priority order):

| # | Approach | How It Works | MPM Code Change? |
|---|----------|-------------|-----------------|
| A | **Prompt injection by team lead** | Team lead includes BASE_AGENT + agent instructions in the Agent tool's `prompt` param when spawning teammates | No — PM_INSTRUCTIONS.md behavioral change only |
| B | **Pre-deploy to .claude/agents/** | MPM deploys agent templates (with BASE_AGENT prepended) to `.claude/agents/`; Claude Code picks them up via `subagent_type` matching | Minor — deployment pipeline change |
| C | **Hook-based injection** | MPM intercepts teammate spawn via hooks and injects context before execution | Requires new hook (may not exist yet) |

**Test protocol:**
- For each approach: spawn a teammate, verify it respects circuit breakers, produces verification evidence, tracks files
- Measure token overhead (acceptable if < 15% of teammate's context window)
- **Pass criteria:** Teammate behaves like an MPM-specialized agent
- **Fail criteria:** Context too large, teammate ignores injected instructions, or unacceptable cost
- **Note:** Approach A is testable immediately since the team lead already has full MPM context and can embed it in prompts

**Deliverable 2: Team Circuit Breaker Protocol**
- Formal document mapping each of the 13 circuit breakers to:
  - "Team-enforced" (each teammate follows individually)
  - "PM-enforced" (PM applies at team boundary on aggregate output)
  - "Not applicable in team context"
- **Pass criteria:** No circuit breaker is left unaddressed
- **Fail criteria:** Any circuit breaker cannot be meaningfully enforced in team context

**Decision gate:** If BOTH deliverables pass -> proceed to Phase 1. If EITHER fails -> shelve integration, improve existing `run_in_background` patterns instead.

### Phase 1: Minimal Viable Integration (Only if Phase 0 passes)
- Add Agent Teams as optional pathway for **parallel research only**
- Strict fallback: if `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` absent, use existing background agents
- PM output format unchanged — team result presented as single research artifact
- No user-visible configuration or decision points

### Phase 2: Expansion (Only if Phase 1 validates in production)
- Expand to parallel QA (requires QA aggregation protocol — HARD)
- Expand to complex features (requires team-lead role definition — HARD)
- Monitor cost vs benefit metrics
- Assess Anthropic API stability signals

### Alternative: Improve Existing Patterns (If Phase 0 fails)
If proof-of-concept fails, invest instead in:
1. Better result aggregation for `run_in_background` agents
2. Shared output directory convention for parallel worktree agents
3. PM-mediated coordination protocol (PM relays messages between background agents)
4. Document existing parallel patterns as "MPM Parallel Agents" pattern

## What Would Make This a Clear "Yes"

- Teammate context injection works cleanly (< 15% token overhead)
- Circuit Breaker Protocol can be formalized without PM complexity increase
- Anthropic signals Agent Teams moving toward stable API
- PM context window relief is measurable (longer sessions before context saturation)
- Per-teammate model selection produces cost savings vs current uniform-model delegation
- A concrete use case demonstrates measurable quality improvement over background delegation

## What Would Make This a Clear "No"

- Teammate context injection proves impossible or unacceptably expensive
- Agent Teams API changes before Phase 0 completes
- Circuit breaker integrity cannot be maintained at team boundary
- PM context saturation is not a real problem in practice (Task tool results don't meaningfully bloat PM context)
- The alternative (enhanced background delegation) proves adequate for all use cases
