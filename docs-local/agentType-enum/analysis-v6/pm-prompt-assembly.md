# PM Prompt Assembly Analysis
## How Agent Names Flow Into the PM System Prompt

**Analyst**: prompt-tracer
**Date**: 2026-03-04
**Branch**: agenttype-enums

---

## Executive Summary

The PM system prompt is assembled by a chain of services: `SystemInstructionsService` → `FrameworkLoader` → `ContentFormatter` + `CapabilityGenerator`. The "Available Agent Capabilities" section is dynamically generated by reading `.claude/agents/*.md` files and extracting their YAML frontmatter `name` field — which becomes the agent "ID" displayed to the PM. This `name` field is what Claude Code's Agent/Task tool uses as the `subagent_type` value. Critically, this creates a **two-identity problem**: each agent has (1) a filename stem (e.g., `research-agent`), (2) a `name` field (e.g., `Research`), and (3) an `agent_type` field (e.g., `research`) — and different subsystems use different ones.

## 1. PM System Prompt Assembly Chain

### 1.1 Entry Point: SystemInstructionsService

`src/claude_mpm/services/system_instructions_service.py`

```
SystemInstructionsService.load_system_instructions()
  └─► FrameworkLoader.get_framework_instructions()
```

- `load_system_instructions()` creates a `FrameworkLoader` and calls `get_framework_instructions()`
- Results are cached in `self._loaded_instructions`
- The `{{AGENT_CAPABILITIES}}` template variable is processed via `process_base_pm_content()` if needed

### 1.2 FrameworkLoader Assembly

`src/claude_mpm/core/framework_loader.py`

```
FrameworkLoader.get_framework_instructions()
  └─► _format_full_framework()  (if agents loaded)
      ├── _generate_agent_capabilities_section()  ← KEY STEP
      ├── context_generator.generate_temporal_user_context()
      └── content_formatter.format_full_framework(
              framework_content,
              capabilities_section,
              context_section,
              ...
          )
```

### 1.3 ContentFormatter Assembly Order

`src/claude_mpm/core/framework/formatters/content_formatter.py`

The PM system prompt is assembled in this order:

1. **INSTRUCTIONS.md** (framework base PM instructions → `PM_INSTRUCTIONS.md`)
2. **Custom INSTRUCTIONS.md** (project-level overrides, if present)
3. **WORKFLOW.md** (workflow phases and protocols)
4. **MEMORY.md** (memory guidelines)
5. **PM Memories** (`~/.claude-mpm/memories/PM.md` actual memories)
6. **`capabilities_section`** ← dynamically generated from deployed agents
7. **Temporal/user context** (date, user info)
8. **BASE_PM.md** (framework requirements appended last)

## 2. Agent Capabilities Section Generation

### 2.1 Discovery Flow

`src/claude_mpm/core/framework_loader.py :: _generate_agent_capabilities_section()`

```
_generate_agent_capabilities_section()
  ├── _discover_local_json_templates()          (local JSON overrides)
  └── for agents_dir in [.claude/agents/, ~/.claude/agents/]:
        for agent_file in agents_dir.glob("*.md"):
          _parse_agent_metadata(agent_file)      → agent_data dict
  └── capability_generator.generate_capabilities_section(
          deployed_agents, local_agents
      )
```

Directories scanned (in order, all merged):
1. `Path.cwd() / ".claude" / "agents"` (project-level, highest priority)
2. `Path.home() / ".claude" / "agents"` (user-level)

### 2.2 Metadata Extraction: The Name → ID Mapping

`src/claude_mpm/core/framework/formatters/capability_generator.py :: parse_agent_metadata()`

```python
# Default: use filename stem
agent_data = {
    "id": agent_file.stem,           # e.g., "research-agent"
    "display_name": agent_file.stem.replace("_", " ").replace("-", " ").title(),
    ...
}

# YAML frontmatter overrides:
if content.startswith("---"):
    metadata = yaml.safe_load(frontmatter)
    # CRITICAL: name field REPLACES the file stem as the ID
    agent_data["id"] = metadata.get("name", agent_data["id"])
    #                  ^^^^^^^^^^^^^^^^^^^
    # "name: Research" → agent_data["id"] = "Research"
    # "name: Engineer" → agent_data["id"] = "Engineer"
```

**This means: the YAML `name` field becomes the agent "ID" displayed to the PM.**

### 2.3 How the PM Sees Agents in Its Prompt

`capability_generator.generate_capabilities_section()` renders:

```markdown
## Available Agent Capabilities

### Research (`Research`)
Memory-efficient codebase analysis...
- **Routing**: Keywords: research, investigate, analyze...

### Engineer (`Engineer`)
Clean architecture specialist...
```

Then appends:
```
Use the agent ID in parentheses when delegating via Task tool.
```

**So the PM is told: use `Research`, `Engineer`, `QA`, etc. as the subagent_type.**

## 3. Agent Name vs ID vs agent_type: The Three-Identity Problem

Each deployed agent has multiple identifiers:

| Field | Example (research agent) | Source | Used By |
|-------|--------------------------|--------|---------|
| Filename stem | `research-agent` | `.claude/agents/research-agent.md` | Claude Code file routing, `AgentLoader.get_deployed_agents()` |
| `name` (frontmatter) | `Research` | YAML `name:` field | `CapabilityGenerator` → displayed to PM as the delegation ID |
| `agent_id` (frontmatter) | `research-agent` | YAML `agent_id:` field | MPM internal tracking |
| `agent_type` (frontmatter) | `research` | YAML `agent_type:` field | `DynamicSkillsGenerator`, `read_agent_type()`, hook handlers |

**Evidence from `.claude/agents/research-agent.md`:**
```yaml
name: Research          # → becomes agent "ID" shown to PM
agent_id: research-agent
agent_type: research    # → used by MPM internals
```

**Evidence from `.claude/agents/engineer.md`:**
```yaml
name: Engineer
agent_id: engineer
agent_type: engineer
```

### 3.1 The Routing Disconnect

The PM sees agents with `name`-based IDs (e.g., `Research`, `Engineer`), but:

**`todo_task_tools.py` (generated CLAUDE.md guidance) says use filename stems:**
```python
# src/claude_mpm/services/framework_claude_md_generator/section_generators/todo_task_tools.py
"subagent_type=\"research-agent\"" # ← file stem, NOT the name field
"subagent_type=\"engineer\""       # ← file stem (happens to match name)
"subagent_type=\"qa-agent\""       # ← file stem
```

**`utility_service.py` (delegation detection) checks for:**
```python
delegation_patterns = [
    "engineer agent",
    "qa agent",
    "research agent",
    ...
]
# Uses "agent_name agent" patterns, not subagent_type values
```

**PM_INSTRUCTIONS.md (the actual PM prompt) references:**
```markdown
## When to Delegate to Each Agent
| **Research** | ...
| **Engineer** | ...
| **Ops** (local-ops) | ...    ← uses "local-ops" specifically
| **QA** (web-qa-agent, api-qa-agent) | ...
```

**WORKFLOW.md routing code:**
```python
if "API" in implementation: use api_qa
elif "UI" in implementation: use web_qa
else: use qa
```
These are Python-style identifiers, not the actual `name` or filename values.

## 4. How Delegation Resolves: Agent Tool Routing

### 4.1 Claude Code's Agent Tool

When PM calls `Agent(subagent_type="Research")`, Claude Code:
1. Looks through all `.claude/agents/*.md` files
2. Matches the `name` field in YAML frontmatter to the `subagent_type` value
3. Loads that agent's instructions as the subagent's system prompt

**This means `name` field IS the canonical delegation identifier.**

### 4.2 The subagent_type → File Resolution

```
subagent_type="Research"
  → Claude Code scans .claude/agents/
  → Finds research-agent.md with name: Research
  → Uses research-agent.md as the agent's instructions
```

But `subagent_type="research-agent"` would also work (filename stem matching as fallback).

### 4.3 What the PM Actually Uses

From the system prompt the PM receives, the "Available Agent Capabilities" section shows:

```
### Research (`Research`)
```

So the PM will call `Agent(subagent_type="Research")` — matching the `name` field.

But the fallback hardcoded capabilities in `ContentFormatter.get_fallback_capabilities()` show:
```
- **Research** (`research-agent`): Investigation and analysis
```
← Uses filename stem, not `name` field!

## 6. The `agent_type` Field's Role

`agent_type` (in YAML frontmatter) is an **MPM-internal categorization field**, NOT the Claude Code routing key.

Used in:
- `DynamicSkillsGenerator._build_agent_skill_content()` → `read_agent_type(agent_info, "general-purpose")`
- `socketio/handlers/hook.py` → `agent_type = data.get("agent_type", "unknown")` for event tracking
- `deployed_agent_discovery.py` → `agent_type = getattr(agent, "agent_type", ...)` for registry
- `SkillManager` → `agent_types=["pm", "project-manager", "project_manager"]` for skill matching

**It does NOT appear in the PM system prompt** (neither in capabilities section nor in routing instructions).

## 8. Key Findings for agenttype-enums Work

### Finding 1: `name` Field Is the Primary PM-Facing Identifier
The PM system prompt displays the YAML `name` field as the agent ID. The `name` field is what the PM uses in `subagent_type=` calls. This is the critical identity for delegation.

### Finding 2: `agent_type` Is NOT in the PM Prompt
The `agent_type` field (e.g., `research`, `engineer`) is purely MPM-internal. Renaming or standardizing `agent_type` values does NOT affect what the PM sees or how delegation works.

### Finding 3: Multiple Hardcoded Naming Conventions Coexist
- `PM_INSTRUCTIONS.md`: Uses display names (`Research`, `local-ops`, `web-qa-agent`)
- `todo_task_tools.py`: Uses filename stems (`research-agent`, `qa-agent`)
- `ContentFormatter` fallback: Mixed (`engineer` vs `research-agent`)
- `AGENT_TEMPLATES` dict: Uses underscore keys (`api_qa`, `web_qa`)

### Finding 4: Fallback Capabilities Have Inconsistent IDs
When no agents are deployed, the hardcoded fallbacks use a mix of identifier styles. This would be shown to PM if `.claude/agents/` is empty, potentially causing wrong `subagent_type` values.

### Finding 5: The `name` Field Must Stay Stable
Since Claude Code routes agent calls by matching `subagent_type` to the `name` field, any rename of the `name` field values would break delegation. The `agent_type` field can be standardized safely without breaking PM routing.

### Finding 6: WORKFLOW.md Uses Role Labels, Not Technical IDs
WORKFLOW.md references agents by role (`Research`, `Code Analyzer`) and by Python-style identifiers (`api_qa`, `web_qa`) that don't match either the `name` or filename. These are loose references interpreted by context.
