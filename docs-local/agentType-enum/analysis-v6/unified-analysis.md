## 1. Executive Summary

The rename proposal standardizes agent template filenames from underscore format (`golang_engineer.md`) to hyphen format (`golang-engineer.md`). The motivation is aesthetic consistency: some files already use hyphens (`python-engineer.md`, `local-ops-agent.md`) while others use underscores. A rename script (`scripts/rename_templates_to_dashes.sh`) exists and targets source templates under `src/claude_mpm/agents/templates/`.

The key question is whether this rename is a cosmetic change or a breaking change. The answer is: **it is a breaking change if not executed atomically across all affected systems simultaneously, and a cosmetic change only for the PM-facing delegation identity** (which is controlled by the YAML frontmatter `name:` field, not the filename stem).

The bottom-line answer: the codebase has at least **4 confirmed breaks** and **3 likely breaks** if the rename is applied to source templates without simultaneously updating (1) `scripts/bump_agent_versions.py`, (2) `src/claude_mpm/services/agent_capabilities_service.py`, (3) `src/claude_mpm/config/agent_capabilities.yaml`, and (4) deployed files in `.claude/agents/`. The system is also already inconsistent before any rename: `.claude/agents/` right now contains a mix of underscore and hyphen filenames. Renaming source templates without resolving the deployed directory amplifies this existing inconsistency.

## 2. How Agent Names Flow Through the System

Source: `pm-prompt-assembly.md`

### 2.1 Assembly Chain

The PM system prompt is assembled by a chain of services:

```
SystemInstructionsService.load_system_instructions()
  └─► FrameworkLoader.get_framework_instructions()
      └─► _format_full_framework()
          ├── _generate_agent_capabilities_section()   ← KEY STEP
          ├── context_generator.generate_temporal_user_context()
          └── content_formatter.format_full_framework(...)
```

The "Available Agent Capabilities" section is generated by `_generate_agent_capabilities_section()` in `src/claude_mpm/core/framework_loader.py`. This function scans two directories:

1. `Path.cwd() / ".claude" / "agents"` (project-level, highest priority)
2. `Path.home() / ".claude" / "agents"` (user-level)

For each `*.md` file found, it calls `CapabilityGenerator.parse_agent_metadata()`.

### 2.2 The Name-to-ID Mapping

`src/claude_mpm/core/framework/formatters/capability_generator.py :: parse_agent_metadata()`

```python
# Default: use filename stem
agent_data = {
    "id": agent_file.stem,   # e.g., "research-agent"
    ...
}

# YAML frontmatter OVERRIDES the default:
if content.startswith("---"):
    metadata = yaml.safe_load(frontmatter)
    agent_data["id"] = metadata.get("name", agent_data["id"])
    #                  ^^^^^^^^^^^^^^^^^^^
    # "name: Research" → agent_data["id"] = "Research"
```

The YAML `name:` field replaces the filename stem as the agent ID shown to the PM. If no frontmatter exists, the filename stem is used as a fallback.

### 2.3 What the PM Sees

The `generate_capabilities_section()` call renders this in the PM prompt:

```markdown
## Available Agent Capabilities

### Research (`Research`)
Memory-efficient codebase analysis...

### Engineer (`Engineer`)
Clean architecture specialist...

Use the agent ID in parentheses when delegating via Task tool.
```

The PM is instructed to use `Research`, `Engineer`, `QA`, etc. as the `subagent_type` value when calling the Agent/Task tool.

### 2.4 How Claude Code Resolves subagent_type

When the PM calls `Agent(subagent_type="Research")`, Claude Code:

1. Scans all `.claude/agents/*.md` files
2. Matches the `subagent_type` value against each file's YAML frontmatter `name:` field
3. Loads the matching file as the subagent's system prompt

Filename stem matching is a fallback. The primary resolution mechanism is frontmatter `name:` field matching.

**Confirmed from deployed agent files:**

`.claude/agents/research-agent.md`:
```yaml
name: Research          # → becomes agent "ID" shown to PM, used in subagent_type=
agent_id: research-agent
agent_type: research    # → MPM internal only
```

`.claude/agents/engineer.md`:
```yaml
name: Engineer
agent_id: engineer
agent_type: engineer
```

### 2.5 The Three-Identity Problem

Each deployed agent carries three distinct identifiers used by different subsystems:

| Field | Example | Source | Used By |
|-------|---------|--------|---------|
| Filename stem | `research-agent` | `.claude/agents/research-agent.md` | Claude Code file routing (fallback), `AgentLoader.get_deployed_agents()` |
| `name` (frontmatter) | `Research` | YAML `name:` field | `CapabilityGenerator` — displayed to PM as the delegation ID |
| `agent_id` (frontmatter) | `research-agent` | YAML `agent_id:` field | MPM internal tracking |
| `agent_type` (frontmatter) | `research` | YAML `agent_type:` field | `DynamicSkillsGenerator`, hook handlers, event tracking |

The `agent_type` field does NOT appear in the PM system prompt. It is purely MPM-internal. Changes to `agent_type` values do not affect PM delegation routing.

### 2.6 Hardcoded References in PM-Facing Files

Multiple files that are injected verbatim into the PM system prompt contain hardcoded agent name strings with inconsistent conventions:

**`src/claude_mpm/agents/PM_INSTRUCTIONS.md`**: Uses `local-ops` (hyphen, no suffix), `web-qa-agent` (hyphen + suffix), and `engineer` (bare).

**`src/claude_mpm/services/framework_claude_md_generator/section_generators/todo_task_tools.py`**: Generates CLAUDE.md content with these `subagent_type` examples:
- `subagent_type="research-agent"` (hyphen + `-agent` suffix)
- `subagent_type="engineer"` (bare — no suffix)
- `subagent_type="qa-agent"` (hyphen + `-agent` suffix)
- Explicitly marks `subagent_type="research"` as ❌ WRONG

This is internally contradictory: `engineer` requires no suffix, but `research` and `qa` require `-agent`.

**`src/claude_mpm/core/framework/formatters/content_formatter.py`** fallback capabilities (shown when no agents are deployed):
```
"- **Engineer** (`engineer`): Code implementation"
"- **Research** (`research-agent`): Investigation"
"- **QA** (`qa-agent`): Testing"
```
Mixed: `engineer` uses bare name, others use filename-style.

## 4. Break Analysis

Source: `devils-advocate.md`

### 4.1 Risk Matrix

| Break | Severity | Detectability | How to Trigger |
|-------|----------|--------------|----------------|
| `bump_agent_versions.py` file path | HIGH | Easy (FileNotFoundError) | Run version bumper script |
| `agent_capabilities_service.py` stem mismatch | HIGH | Silent (wrong data) | Any agent listing/capabilities query after redeploy |
| YAML internal inconsistency amplification | MEDIUM | Silent | Agent auto-configuration feature |
| `tool_analysis.py` exact string matches | MEDIUM | Silent | Dashboard delegation tracking |
| `subagent_processor.py` list check | LOW | Silent | Dashboard event filtering |
| Duplicate agents in `.claude/agents/` | HIGH | Intermittent | Redeploy without cleanup of old files |
| User scripts with hardcoded names | HIGH | Silent until run | User automation, CI/CD scripts |
| Memory file naming mismatch | MEDIUM | Silent (lost memories) | Agent memory lookups after rename |

### 4.2 Confirmed Breaks

**Break 1: `scripts/bump_agent_versions.py` — FileNotFoundError**

Evidence: `scripts/bump_agent_versions.py` lines 10-42 hardcode underscore names to construct file paths:
```python
AGENTS_WITH_SKILLS = [
    "golang_engineer",
    "local_ops_agent",
    "python_engineer",
    ...
]
```
After rename: `AGENTS_DIR / f"{agent_name}.json"` constructs a path to a file that no longer exists. The script throws `FileNotFoundError` or silently skips every agent. Additionally, the script already contains `"php-engineer"` and `"ruby-engineer"` (hyphens) mixed with the underscore names — evidence of a partially-applied rename that created the bug this analysis warns about.

**Break 2: `agent_capabilities_service.py` — Silent Wrong Data**

Evidence: `src/claude_mpm/services/agent_capabilities_service.py` line 224:
```python
agent_id = agent_file.stem  # NO NORMALIZATION
```
Line 248:
```python
discovered_agents[agent_id] = {
    "id": agent_id,
    "category": self._categorize_agent(agent_id, content),
}
```
After redeploy with renamed files: `agent_id = "golang-engineer"` (hyphen from stem). But `agent_capabilities.yaml` has key `golang_engineer` (underscore). The lookup `agent_capabilities["golang-engineer"]` misses. The agent gets wrong/empty category and metadata. This is a silent failure — no exception is raised.

Contrast: `unified_agent_registry.py` line 319 DOES normalize:
```python
return name.lower().replace("-", "_").replace(" ", "_")
```
Two discovery paths produce different IDs for the same agent.

**Break 3: `agent_capabilities.yaml` Inconsistency Amplification**

Evidence: `src/claude_mpm/config/agent_capabilities.yaml` lines 222-259:
```yaml
php_engineer:                    # outer key: underscore
  agent_id: "php-engineer"       # inner value: hyphen — ALREADY INCONSISTENT

ruby_engineer:                   # outer key: underscore
  agent_id: "ruby-engineer"      # inner value: hyphen — ALREADY INCONSISTENT

python_engineer:                 # outer key: underscore
  agent_id: "python_engineer"    # inner value: underscore — consistent

local_ops_agent:
  agent_id: "local_ops_agent"
  metadata:
    template_file: "local-ops.md"  # ALREADY references hyphen target
```
The rename must decide: update the YAML outer keys (breaking all Python code that does `yaml_data["golang_engineer"]`) or leave them (YAML remains internally inconsistent). Both options have downstream breaks.

**Break 4: Duplicate Agent Files After Redeploy**

Evidence: Current `.claude/agents/` (as of branch `agenttype-enums`) contains:
- Underscore files: `golang_engineer.md`, `java_engineer.md`, `ruby_engineer.md`, `php_engineer.md`, etc.
- Hyphen files: `local-ops-agent.md`, `python-engineer.md`, `typescript-engineer.md`, etc.

After source template rename but before redeploy: source templates all use hyphens, deployed files remain mixed. After redeploy: deployment copies hyphen-named templates to `.claude/agents/`. Now both `golang-engineer.md` AND `golang_engineer.md` exist simultaneously. Framework discovery may pick either, or both fire, causing double-execution.

### 4.3 Likely Breaks

**Break 5: `tool_analysis.py` Exact String Matching**

Evidence: `src/claude_mpm/hooks/claude_hooks/tool_analysis.py` lines 84-86:
```python
"is_pm_delegation": tool_input.get("subagent_type") == "pm",
"is_research_delegation": tool_input.get("subagent_type") == "research",

The dashboard incorrectly marks delegations as non-delegation events.

**Break 6: `subagent_processor.py` Hardcoded List**

Evidence: `src/claude_mpm/hooks/claude_hooks/services/subagent_processor.py` lines 351-352:
```python
"is_delegation_related": agent_type

```
If `agent_type` is derived from a renamed file stem (e.g., `"local-ops"` instead of `"local_ops"`), agents not on this list are misclassified. Affects monitoring and dashboard accuracy.

**Break 7: `AgentNameNormalizer` Partial Coverage**

Evidence: `src/claude_mpm/core/agent_name_normalizer.py` line 283:
```python

## 5. Critical Unanswered Question

**Does Claude Code resolve `subagent_type` from the filename stem or from the frontmatter `name:` field?**

This is the pivotal question for determining whether the rename is a cosmetic change or a breaking change for Task tool routing.

**Evidence from the PM prompt assembly analysis** (pm-prompt-assembly.md, Finding 1):
The `CapabilityGenerator.parse_agent_metadata()` function reads the YAML frontmatter `name:` field and uses it as the agent ID shown to the PM. The PM is explicitly told: "Use the agent ID in parentheses when delegating via Task tool." The ID in parentheses is the `name:` field value (e.g., `Research`, `Engineer`).

**Evidence from deployed agent files**:
`research-agent.md` has `name: Research`. The PM calls `Agent(subagent_type="Research")`.

**What the pm-prompt-assembly.md analysis found** (section 4.2):
```
subagent_type="Research"
  → Claude Code scans .claude/agents/
  → Finds research-agent.md with name: Research
  → Uses research-agent.md as the agent's instructions
```
The analysis states that `subagent_type="research-agent"` would also work as a filename stem fallback.

**Why this question remains unanswered**:
The pm-prompt-assembly.md analysis is based on reading the MPM source code and inferring Claude Code's behavior. It was not validated by empirically testing what happens when `subagent_type` values and file stems diverge. The analysis states "But `subagent_type='research-agent'` would also work (filename stem matching as fallback)" — this is an inference, not a verified observation.

**The stakes**: If Claude Code resolves `subagent_type` primarily from the frontmatter `name:` field, then renaming files from `golang_engineer.md` to `golang-engineer.md` is cosmetically safe as long as the frontmatter `name:` field value does not change. If Claude Code resolves `subagent_type` from the filename stem first, then any code that passes `subagent_type="golang_engineer"` will break immediately after the rename.

**Recommended action**: Deploy a test agent with a deliberate mismatch (filename stem differs from frontmatter `name:`), then observe which value Claude Code uses for routing. This can be done in under 30 minutes and eliminates the primary uncertainty.

v5's risk rating of the five collision pairs as CRITICAL / CERTAIN was correct. v6 confirms these pairs still exist on the `agenttype-enums` branch. v5's identification of 7 unnormalized deployment paths was accurate. v5's analysis of `ensure_agent_id_in_frontmatter()` behavior was confirmed by reading the same source code in v6.

### 7.4 What v5 Assessed Incorrectly or Incompletely

v5 treated all agent name changes as equally risky for delegation routing. v6's PM prompt assembly analysis shows that delegation routing uses the `name:` frontmatter field, not the filename stem. This means the rename's impact on delegation is less severe than v5 suggested — provided `name:` fields remain unchanged.

v5 did not distinguish between changes to `agent_type` (MPM-internal, safe to change) and changes to `name:` (PM-facing, breaks delegation if changed).

## 9. Open Questions

**Q1 (Critical): Claude Code subagent_type resolution mechanism**

Does Claude Code resolve `subagent_type` by matching against the filename stem of `.claude/agents/*.md` files, or by matching against the YAML frontmatter `name:` field? The pm-prompt-assembly.md analysis infers that `name:` field is primary, but this has not been empirically validated. Resolution: deploy a test agent with deliberate mismatch and observe behavior.

**Q2 (High): What is the canonical agent name format?**

The project has two conflicting normalization modules that produce different canonical forms. This conflict predates the rename proposal. Before executing the rename, a decision must be made and both modules updated. Options: (a) underscore canonical, matching `agent_name_normalizer.py`; (b) hyphen canonical, matching `agent_registry.py` and `skill_to_agent_mapping.yaml`.

**Q3 (High): What happens to the 15 `-agent` suffix files during normalization?**

`normalize_deployment_filename()` strips the `-agent` suffix as well as converting underscores to hyphens. This affects 15 files in the deployed directory and produces 5 collisions. The plan documents only the underscore-to-hyphen conversion. The `-agent` suffix stripping behavior must be explicitly addressed.

**Q4 (Medium): Are memory files named after agent IDs?**

Agent memory files in `.claude-mpm/memories/` may be named after agent IDs (e.g., `golang_engineer.md`). If so, renaming the agent's effective ID means the agent can no longer find its own memories. The naming convention for memory files needs to be investigated.

**Q5 (Medium): What is the correct `subagent_type` for `local-ops`?**

`PM_INSTRUCTIONS.md` uses `local-ops`. `pm-examples.md` uses `local-ops-agent`. The deployed file is named `local-ops-agent.md`. The `name:` frontmatter field would determine what the PM actually passes. This specific agent has the most naming variants across all files and needs explicit resolution.

**Q6 (Low): Are there additional `-agent` suffix files in the remote cache?**

The `~/.claude-mpm/cache/agents/` directory was inaccessible during analysis. The remote cache may contain additional files with underscore or `-agent` names that would be deployed to `.claude/agents/` on next sync, re-creating the inconsistency.
